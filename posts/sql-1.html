SQL Language — Complete Practical Handbook

1. The Example Database Used Everywhere

To keep all examples consistent, we will use the same imaginary database:

customers(id, name, email, city)
orders(id, customer_id, order_date, total)
order_items(id, order_id, product, quantity, price)
employees(id, name, department, salary)

You will see these tables in every example so the logic builds naturally.

2. What SQL Really Is

SQL is declarative. You do not tell the database how to do something. You tell it what you want, and the engine decides the best way.

SELECT name FROM customers;
You are describing a result, not writing a procedure.

3. The Most Important Thing Beginners Don’t Know — Execution Order

FROM
JOIN
WHERE
GROUP BY
HAVING
SELECT
ORDER BY

This explains many “mysterious” errors.

You cannot use aliases in WHERE because SELECT has not run yet.

4. SELECT Is Not Just “Get Columns”

SELECT name,
       salary,
       salary * 12 AS yearly_salary
FROM employees;

You can compute, rename, and transform data.

Trying this:
SELECT salary * 12 AS yearly
FROM employees
WHERE yearly > 50000;
Fails because WHERE runs before SELECT.

5. WHERE — Row Filtering Logic

SELECT *
FROM employees
WHERE salary > 5000
AND department IN ('IT','HR')
AND name LIKE 'A%';

WHERE always filters individual rows before any grouping happens.

6. JOINs — How Tables Talk to Each Other

INNER JOIN

SELECT c.name, o.total
FROM customers c
INNER JOIN orders o
ON c.id = o.customer_id;

LEFT JOIN

SELECT c.name, o.total
FROM customers c
LEFT JOIN orders o
ON c.id = o.customer_id;
LEFT JOIN keeps customers even if they never ordered.
Forgetting the ON condition creates a Cartesian product (huge accidental dataset).

7. GROUP BY — Turning Rows Into Groups

SELECT department, COUNT(*), AVG(salary)
FROM employees
GROUP BY department;
This is invalid in most SQL engines: 
SELECT department, name
FROM employees
GROUP BY department;
Because name is not grouped or aggregated.

8. HAVING — Filtering Groups

SELECT department, AVG(salary)
FROM employees
GROUP BY department
HAVING AVG(salary) > 5000;

HAVING is like WHERE but for groups.

9. Subqueries — Query Inside a Query

SELECT name
FROM employees
WHERE salary > (
    SELECT AVG(salary) FROM employees
);

Correlated Subquery

SELECT name
FROM employees e
WHERE salary > (
    SELECT AVG(salary)
    FROM employees
    WHERE department = e.department
);

10. CTE (WITH) — Cleaner Than Subqueries

WITH big_orders AS (
    SELECT * FROM orders WHERE total > 1000
)
SELECT * FROM big_orders;

11. Window Functions — Advanced Analytics Without Grouping

SELECT name, department, salary,
RANK() OVER (PARTITION BY department ORDER BY salary DESC) AS rank_in_dept
FROM employees;

Rows are not collapsed like GROUP BY.

12. Indexes — Why Some Queries Are Fast

CREATE INDEX idx_customer ON orders(customer_id);
Indexes speed up WHERE and JOIN but slow down INSERT and UPDATE.

13. Transactions and ACID

BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
UPDATE accounts SET balance = balance + 100 WHERE id = 2;
COMMIT;

Atomicity, Consistency, Isolation, Durability.

14. Normalization — Good Table Design

1NF: Atomic values
2NF: No partial dependency
3NF: No transitive dependency

15. Constraints — Protecting Data Integrity

CREATE TABLE users(
 id INT PRIMARY KEY,
 email VARCHAR(100) UNIQUE,
 age INT CHECK(age >= 18),
 customer_id INT,
 FOREIGN KEY (customer_id) REFERENCES customers(id)
);

16. Real Use Case Example

“Find customers who spent more than $5000 total”

SELECT c.name, SUM(o.total) AS spent
FROM customers c
JOIN orders o ON c.id = o.customer_id
GROUP BY c.name
HAVING SUM(o.total) > 5000;

17. Another Use Case

“Find the highest paid employee in each department”

SELECT *
FROM (
    SELECT name, department, salary,
    RANK() OVER (PARTITION BY department ORDER BY salary DESC) r
    FROM employees
) t
WHERE r = 1;
